## はじめに

AI 協働開発を続けるうちに、自分の中でひとつの仮説が育ってきました。

**「AI 協働開発では、理解よりも判断の比重を上げる方が、ビジネスとしては合理的なのではないか」**

これまでの自分は「判断 2 : 理解 8」の割合で仕事を進めてきました。コードを一行ずつ読み切ってから判断する、という進め方です。しかし AI が実装の主力になると、人間がすべてを理解しきるコストはスピードに見合わなくなります。

そこで「判断 8 : 理解 2」に振り切ってみたい。ただしそのとき、**理解していない部分の品質をどう担保するか** が問題になります。本記事はその問いに対して、個人プロジェクトでの試行錯誤から見えてきた現時点の答えをまとめたものです。

検証の舞台にしたのは、3 システム構成（React + Vue/Laravel + Spring Boot）の EC を個人で運用している [Amazia プロジェクト](https://github.com/Tetsu0222/amazia) です。

## 「判断 8 : 理解 2」が抱える根本問題

理解を捨てた瞬間に発生する問題は、シンプルです。

- **AI が出してきた実装が、なぜ動くのか分からない**
- **AI が踏みやすい落とし穴に、人間も気づけない**
- **トラブルが起きたとき、根本原因を追えない**

「分からないけど動いている」を放置すれば技術的負債が雪だるま式に増えていく。これは AI 以前から言われている話で、目新しさはありません。

ただ AI 協働の場合、もう一段やっかいな性質があります。AI は **「閉じた文脈」では人間のトッププロを超える出力を出す** 一方で、**文脈が開いていると平気で誤った前提を作り出す**。チェスや将棋のように勝利条件・駒の動き・ロールが明確な領域では強いのに、業務ソフトウェア開発のように暗黙の前提が散らばっている領域では、平然と存在しないライブラリを呼び出したり、既存のスキーマを読まずに矛盾するテーブルを足したりします。

つまり「判断 8 : 理解 2」を成立させるには、AI に渡す文脈そのものを **閉じる** 努力が必要になる。これが本プロジェクトでのスタート地点でした。

## 採った方法論：ドキュメント駆動で文脈を閉じる

具体的には、AI に渡すコンテキストを 3 層構造で設計しました。

```
第1層：CLAUDE.md           ── AI への運用ルール
       ↓ 参照
第2層：docs/ai_context/    ── 知見ライブラリ
       ↓ 出典
第3層：docs/troubles/      ── 一次記録（50 件超の実トラブル）
```

ポイントは、**第 3 層から第 2 層への「昇華」プロセスを CLAUDE.md でルール化している** ことです。

不具合が起きたら、コード修正だけで終わらせない。`docs/troubles/NNN_*.md` に「症状 / 根本原因 / 再発防止 / AI 協働観点」で記録し、そこから抽出できる一般化された落とし穴を `docs/ai_context/ai_collaboration_antipatterns.md`（AP-001〜009）に昇格させる。次のフェーズで AI が計画段階で必ず該当 AP を読むので、同じ穴に二度落ちなくなる。

この「トラブル → 知見化 → AI 参照」のループを、CLAUDE.md に **強制ルール** として書き込んでいます。AI の善意や記憶力に期待しない、というのが大事なところです。

## アンチパターンの「型」を持つ効果

実際に蓄積された AP の一覧はこんな感じです。

| 番号 | アンチパターン |
|---|---|
| AP-001 | 既存スキーマ・既存DDLを読まずに新規追記 |
| AP-002 | Entity / 制約を読まずにテストヘルパーをハードコード |
| AP-003 | 存在しない技術スタックを仮定（Flyway 誤認） |
| AP-004 | API/SDK のシグネチャ・仕様を確認せず外挿 |
| AP-005 | シェル / SQL のクォート・エスケープを読まず実装 |
| AP-006 | 対症療法で本質根因を残す |
| AP-007 | 設計書の機能を部分実装で完了報告 |
| AP-008 | CI/CD・運用設計の検証不足で副次トラブル誘発 |
| AP-009 | テスト分離不足 + 単発 PR で類似クラス見落とし |

これらは抽象的な教訓集ではなく、**それぞれが特定のトラブル番号と紐付いた「出典付きの落とし穴」** です。さらに各 AP には対応する **プロンプトテンプレート（TPL-001〜009）** が双方向リンクで紐付いていて、「DDL を触る作業に着手する → TPL-001 を呼び出す → AP-001 を踏まないチェックリストが AI に渡る」という流れが組まれています。

副次効果として、人間側にも変化がありました。新しいトラブルが起きたとき、「これは AP-006 系（対症療法）の派生か、それとも新規 AP か」という判定が即座にできるようになる。語彙が定まると、思考が速くなります。

## ガードレールはルールだけでなく仕組みで

ただ、CLAUDE.md に「DB/API を変更したらフェーズ内で必ず設計書を更新せよ」と書くだけでは不十分でした。実際、P12〜P14 のフェーズで **14 テーブル / 29 エンドポイントの設計書漏れ** が後から判明しています。

そこで現状は、フェーズ完了の判定条件に「設計書反映の確認」を組み込み、AP-007（設計書の機能を部分実装で完了報告）の対応 TPL で「completion 宣言前に設計書 diff を出力させる」というステップを噛ませています。仕組みでガードする発想 — 個人の注意力に頼らない設計 — は、AI 協働だからこそ効きます。AI も人間も、同じガードレールに守られる関係性になります。

## 振り切ってみて、いま感じていること

ここまで踏み込んでみて、現時点での所感です。

少なくとも本プロジェクトのような規模感 — 3 システム構成・関係者は実質一人 — においては、この進め方は機能していると感じます。「閉じた文脈」を維持できているフェーズでは、AI が出す実装の精度は明らかに上がっていて、自分が一行ずつ読まなくても破綻しない範囲は想定より広がりました。

一方で、規模が大きくなると **AI 自身もドキュメントの全容を把握しきれなくなる** 場面が出てきます。そうなったとき、「ドキュメントを正本としてコードを律する」だけでは不十分で、**ドキュメント自体に対するテスト** — 設計書とコードの静的整合性、同一プロンプトに対する AI 出力の冪等性、過去トラブルへのガードレール網羅性 — という次の問いが立ち上がってきます。

これは N=1 の自己検証で、大規模プロジェクトで実証したわけではありません。ただ「ここまでは手応えがあった、次にはこういう問いがある」というメモとして、現時点の景色を残しておきます。

## おわりに

「理解より判断」に振り切るということは、人間の不安をどう仕組みで肩代わりするか、という設計問題でした。次の記事では、AP-001〜009 をひとつずつ掘り下げて、具体的にどんなトラブルから抽出されたパターンなのかを書いていく予定です。
