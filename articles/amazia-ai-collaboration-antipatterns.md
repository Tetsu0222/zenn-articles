## はじめに

前回の記事 [AI協働で「理解より判断」に振り切ったとき、品質をどう保つか](https://zenn.dev/tetsu0222/articles/amazia-ai-context-overview) で、AI 協働開発において「判断 8 : 理解 2」に振り切ったときの品質担保の仕組みを紹介しました。その中核にあるのが、本記事で扱う **AI協働アンチパターン集（AP-001〜009）** です。

50 件超の実トラブルを記録した [docs/troubles/](https://github.com/Tetsu0222/amazia/tree/main/docs/troubles) から、AI 任せにすると繰り返し踏みやすい落とし穴を 9 つに昇華しました。1 つひとつが特定のトラブル番号と紐付いた **出典付きアンチパターン** です。

本記事は、それを「ひとつひとつ」紹介していくカタログです。同じ穴に落ちた人、これから AI 協働を始める人に向けて、地雷原のマップとして使ってもらえれば。

「分類して並べる」ことの効果は思った以上に大きく、新しいトラブルが起きたとき **「これは AP-006 系の派生か、それとも新規パターンか」と即座に語彙化できる** ようになります。AI に渡すときも「今回の作業は AP-001 を踏みやすい」と前置きできるので、計画段階での精度が変わります。

それでは 1 つずつ見ていきます。

## AP-001: 既存スキーマを読まずに新規 DDL を追記

**症状:** Core 起動時の `schema.sql` 実行で、既存の `users.id`（BIGINT UNSIGNED）と新規テーブルの FK 列（signed BIGINT）の型不一致により `operation_logs` テーブル作成が失敗。`continue-on-error=true` で WARN として握りつぶされ、本番 MySQL に該当テーブルが存在しないまま稼働、API 呼び出し時に 500 エラーが連発しました。

**なぜ AI が踏みやすいか:** H2 テスト環境では UNSIGNED の概念がないため、FK 互換性チェックがすり抜けます。AI は H2 でテストが通れば「OK」と判断しがちで、本番固有の型不整合は CI でも CD でも検知できない。さらに `continue-on-error` が DDL 失敗を WARN に格下げするため、起動ログを丁寧に見ない限り気付けません。

**予防のレバー:** 新規テーブル追加時に「参照先となる既存テーブル（特に `users.id`）の型を必ず確認する」ことを、プロンプトテンプレート（TPL-001）に組み込みました。さらに事後検知として、`ops/healthcheck/required_tables.txt` と本番 MySQL の `information_schema.TABLES` を照合する仕組みを CD に追加しています。「DDL 失敗を WARN で潰さない」のがあるべき姿ですが、運用ガード（事後検知）も二重に張ることで、人間の注意力に頼らない構造にしました。

## AP-002: Entity 制約を読まずにテストヘルパーをハードコード

**症状:** `ProductSku` の `@UniqueConstraint(product_id, color, size)` を確認せず、テストヘルパーで color/size をハードコード値に固定。同一 product に複数 SKU を作成するテストで一斉に制約違反となり、関連テストが全滅しました。

**なぜ AI が踏みやすいか:** テストヘルパーは「動けば良い」と思われがちで、Entity の制約注釈まで読み込まずに書かれることが多い。さらに H2 が JPA Entity 定義から UNIQUE 制約を自動生成するため、手元で 1 回テストを書いた時点では問題が出ません。**他のテストが同じヘルパーを使い始めた瞬間に爆発する** ── 影響範囲が広いのも特徴です。

**予防のレバー:** `test_insights.md` に「`@UniqueConstraint` を持つ Entity のヘルパーは制約カラムを引数化、または `nanoTime` で一意化する」ことを観点として明文化。TPL-003（テストヘルパー作成時）で、Entity を読んで制約注釈を報告させてから実装に入る運用に変更しました。テストヘルパーは「実装より制約に縛られている」と認識を改めるのがポイントです。

## AP-003: 存在しない技術スタックを仮定（Flyway 誤認）

**症状:** Flyway を使用していないプロジェクトで、`db/migration/` ディレクトリが存在しているだけで「Flyway 利用中」と AI が誤認。`V6〜V11.sql` を作成したが本番では実行されず、必要テーブル 6 個が作成されないまま phase14 の注文確定 API が 500 エラーを多発させました。

**なぜ AI が踏みやすいか:** 「`db/migration/` がある = Flyway」というのは、世間一般のプロジェクトでは正しい連想です。AI はこの種の **慣習的連想** に強い。逆に言えば「慣習から外れた構成」を見抜くのが弱い。実際のスキーマ管理は `schema.sql` + `spring.sql.init.mode=always` で行われていたのですが、AI は `pom.xml` の dependency を確認せず、ディレクトリ構造の外見だけで判断しました。

**予防のレバー:** 新フェーズ着手時に「マイグレーション機構の確定」を最初のステップとして要求。`pom.xml` の dependency と起動ログの両方を AI に確認させて、現在の機構を明文化させる。`db/migration/V*.sql` には「本番未実行・名残ファイル」を示す README を置いて、後続の AI/人間が同じ誤認をしないようにしました。**慣習的連想を疑う仕組み** が必要です。

## AP-004: API/SDK のシグネチャを確認せず外挿

**症状:** Market の Checkout 画面で、Core API レスポンスの getter 名（`skuId`）と異なる仮定フィールド（`id`）を参照したため、`sku_id=undefined` で遷移失敗。同型は Console/Core 間でも頻発（Guzzle CookieJar デフォルト無効の見落とし、JJWT の鍵長制約見落としなど）しました。

**なぜ AI が踏みやすいか:** AI は「リスト API と詳細 API は同じ DTO 構造」「ライブラリは推測どおりのデフォルト挙動」といった **常識的な省略** を働かせがちです。普段はそれが効率を生むのですが、システム境界（フロント ⇄ バック、ライブラリ ⇄ アプリ）では裏切られやすい。しかもこの種のミスは **手動 E2E テスト** でしか検知できないことが多く、CI のフロントテストはモックを使うため検知不可です。

**予防のレバー:** TPL-002（ライブラリ・API 採用時）で「該当 DTO クラスの全フィールド名 / リスト API と詳細 API のフィールド差分 / ライブラリの公式ドキュメント引用」を必ず報告させてから実装に進む運用に。さらにフェーズ完了の判定に「実機ブラウザでの貫通テスト」を必須化しました。**境界をまたぐ実装は、推測ではなく引用で進める** のが鉄則です。

## AP-005: シェル / SQL のクォート・エスケープを読まず実装

**症状:** CD パイプラインのヘルスチェック SQL が `docker exec ... sh -c '<INNER>'` の内側にシングルクォート非エスケープで埋め込まれ、SQL リテラルの `'` が外側を閉じてしまい、MySQL が「`amazia` はカラム名」と解釈して Unknown column エラー。直前の修正（046: ホスト側 `$MYSQL_ROOT_PASSWORD` 展開漏れ）の **直後に同類別の問題で再失敗** しました。

**なぜ AI が踏みやすいか:** 複層のクォート展開（ホスト bash → コンテナ sh → SQL リテラル）は、AI が苦手とする領域です。各層で何が展開されるかを順番に追わないと正解にたどり着けないのに、AI は「ぱっと見動きそうなコマンド」を出してしまう。さらに、この種のバグは **本番 SSM RunShellScript 固有の挙動** でしかフルに再現できず、ローカルや CI ですり抜けます。

**予防のレバー:** TPL-005（シェル / SQL 文字列構築時）で「ホスト bash → コンテナ sh の各層で何が展開されるか」を整理させてからコマンドを提示させる運用に。さらに、CD ステップで `set -x` 相当のシェル展開結果ダンプを残し、実際に流れた SQL をログで確認できるようにしました。**故意失敗テストデプロイ**（schema.sql から 1 テーブル削除して `exit 1` を確認する）を PR マージ前に必須化したのも効きました。

## AP-006: 対症療法で本質根因を残す

**症状:** 公開期間判定が秒単位（LocalDateTime）と日付単位（LocalDate JST 0:00）の二重基準のまま温存され、Market では「予約可」だが Checkout 確定時は「公開日未到達」で 400 エラー。020（Cookie domain 上書き忘れ）や 030（DuckDNS 単一ドメイン問題）も同型で、「動いた箇所だけ直して根因を残した」典型例です。

**なぜ AI が踏みやすいか:** AI は **「症状解消 = 解決」** とみなしがちです。同じ概念のロジックが複数 Service / Entity に分散している場合、目の前の不具合だけ直して「修正完了」と報告してしまう。横展開確認をしないため、同根バグが時間差で他箇所に湧き出します。これが個人開発だと、トラブル番号だけが増えて疲弊する原因になります。

**予防のレバー:** TPL-006（バグ修正時の横展開確認）で「同概念のロジックが他のファイルにないか必ず grep で確認」「修正完了の判断基準を『症状解消』ではなく『根因の単一化・横展開確認完了』に置く」ことを義務化。さらに修正後は「直した箇所」と「直さなかった箇所の根拠」を併記して報告させる運用にしました。**「直さなかった箇所」を明示する** だけで、AI の思考プロセスがガラッと変わります。

## AP-007: 設計書の機能を部分実装で完了報告

**症状:** phase10 の設計書「SKU 価格管理機能」はバックエンド API / DB / Service が完全実装されているのに、Vue コンポーネント（`SkuPriceList.vue`）がプレースホルダーのまま残置。画面を開くと「フェーズ10で実装予定です」という空テンプレートが表示。011 / 013 / 039 も同型で、フェーズ完了の定義が曖昧だったことが共通根です。

**なぜ AI が踏みやすいか:** AI は「API が実装された = 機能完了」と判断しがちです。CI が緑になればフェーズ完了と誤認し、ブラウザでの実機確認をスキップする。背景には「設計書の機能全体」と「実装層の分業（フロント / バック / DB）」を切り分ける概念がないことがあります。

**予防のレバー:** フェーズ完了の定義を **「CI 緑 + 実機ブラウザでの全画面操作確認」** と明示し、TPL-007（フェーズ完了確認時）で対応。プレースホルダーコンポーネントには TODO コメント + フェーズ番号を必須化し、フェーズ完了時に grep で残置検出する運用に。設計書には「フロント / バック / DB の実装期限」対応表を持たせて、部分実装での完了宣言を構造的に防ぐようにしました。

## AP-008: CI/CD・運用機構の検証不足で副次トラブル誘発

**症状:** SSM コマンド配信が PingStatus=Online でも Undeliverable になる **「ゾンビ Online」現象** が発生。既存リカバリ機構（PingStatus チェック → stop/start）では検知できず、複数回デプロイ失敗。同根は 003（--build 除去後の検証漏れ）・023（--remove-orphans 漏れ）・008（restart ポリシー漏れ）・025（リカバリ後 Pending 滞留）でも繰り返されました。

**なぜ AI が踏みやすいか:** CI/CD や運用機構は **「何を検証していて、何を保証していないか」** が暗黙のまま実装されがちです。PingStatus=Online は「Agent が応答した」だけで「コマンド配信パスが健全」を意味しないのに、AI は「Online = OK」と短絡しがち。さらに本番 SSM の挙動は本番でしか再現できないため、CI 検証ではすり抜けます。

**予防のレバー:** TPL-008（CI/CD・運用機構導入時）で「この機構が何を検証していて、何を保証していないかを明文化」「関連する過去トラブル（003/008/023/025/026）を引用」「カナリア戦略の有無」を必ず報告させる運用に。**「動いた」を解決とみなさず、保証範囲を明示してから実装する** という姿勢の徹底です。実際の対策としては、軽量 echo コマンドを事前に流す **カナリア方式** で実配信可否を確認するようにしました。

## AP-009: テスト分離不足 + 単発 PR で類似クラス見落とし

**症状:** H2 共有 DB で件数アサーションが失敗。`ApplyScheduledPricesJobTest.APP_3` の単発失敗から始まり、**1 クラスずつ `@Transactional` を付与する対症療法を 4 段階繰り返した結果、最終的に 46 クラスへ一括付与し 6 クラスは設計上 revert** という大手術に発展しました。`@Transactional(REQUIRES_NEW)` 経由の書き込みはロールバックを貫通するため、件数アサーションを行う Repository テストへ自衛コードが増殖し、「テストを通すこと」が「不具合を検知すること」を浸食しました。

**なぜ AI が踏みやすいか:** AP-006（対症療法）と AP-008（検証不足）の **複合形** です。1 件目を見たときに同型汚染を grep せず、対症療法の連鎖に入る。さらに `@SpringBootTest` の H2 共有 DB は **単体実行では PASS、全体実行でだけ失敗** という最も気付きにくい形で顕在化します。

**予防のレバー:** `test_insights.md` に「`@Transactional` を付ける/付けない判断軸」「件数アサーション規約」「スナップショット差分規約」を明文化。`cleanup.sql + @Sql` 方式を規約化し、`@Transactional` を敢えて付けない場合はクラス Javadoc 冒頭に理由を明記する運用に。さらに **週次 cron で `mvn test -Dsurefire.runOrder=random` を実行・失敗時 Issue 自動起票** する仕組みを入れて、テスト順序依存のバグを早期検知するようにしました。最終的には Testcontainers per-class MySQL に移行する判断に。**症状療法の限界を察知して、構造的対策にフェーズを切る** 判断こそが、人間に求められる役割でした。

## AP集を運用してみてわかったこと

9 つを揃えてから半年ほど運用してきて、いくつか手応えがあります。

**1. 新規トラブルの「分類」が即座にできるようになった**
不具合が出たときに「これは AP-006 の派生だな」と語彙化できる。Claude Code に「AP-006 の派生として記録して」と渡すだけで、対応プロンプトスニペットも自動的に呼び出される。**思考の入り口が定型化される** のは大きい。

**2. 「同根トラブル」の連鎖が止まった**
AP-001 を明示した後、users.id 参照系の同型トラブル（045 → 049）は事前に grep で抽出して同時修正するようになりました。**単発バグを単発で閉じない**、という運用が定着した瞬間です。

**3. AI 自身も AP 集を読みに来るようになった**
CLAUDE.md に「実装着手前に該当 AP の対応プロンプトスニペットを確認」と書いたことで、Claude が自発的に AP 集を読み込み、自己チェックリストとして使うようになりました。ガードレールがコンテキストに組み込まれている感覚です。

一方で課題もあります。AP は 9 個に増えてきたあたりから **AI 自身も全容を把握しきれなくなる** 兆候が出始めていて、「どの AP がこの作業に該当するか」の判定が AI の察しに依存しています。今後は **ファイルパターンから TPL を機械的に引ける判定木** のような、決定論的なトリガーが必要になりそうです。

## おわりに

アンチパターン集は「失敗の記録」ではなく **「次に同じ穴に落ちないための前借りの判断材料」** です。50 件超のトラブルを 9 つに凝縮した本記事が、AI 協働開発で同じ穴に落ちかけている方の参考になれば。

次の記事では、各 AP に対応する **プロンプトテンプレート（TPL-001〜009）** の運用 ── 「プロンプトを毎回考える」から「テンプレートを呼び出す」への転換 ── について書こうと思います。
